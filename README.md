# PlayerTicker (Fantasy-Market)

A fantasy football **market and valuation platform** — it tracks player value like a stock ticker so
managers can spot risers, fallers, buy-low windows, and overheated assets before their league reacts.

The repository contains two layers that are deliberately kept separate:

1. **The market terminal app** — the MVP / P0 build described in [`DESIGN.md`](./DESIGN.md): a
   mobile-first, dark-mode SPA running on a clearly-labeled **Demo Market**, a deterministic, seeded
   mock-data engine that simulates daily value movement for ~140 fantasy-relevant players. No login,
   no backend. Live **Sleeper player metadata** (names, teams, status, trending counts) can be
   overlaid on top, but it never changes a price.
2. **Deterministic position valuation engines** — pure, spec-driven TypeScript engines
   (`evaluateWideReceiver`, `evaluateRunningBack`, `evaluateTightEnd`) with golden-fixture test
   suites. The WR and RB engines power the `/player-model` UI; the TE engine is implemented and
   tested but not yet wired into the UI.

> **Demo Market — simulated player values for product preview. Not current player information.**
> Every market value on the site is generated by the internal market engine from authored inputs.

## Current capabilities (verified on this branch)

- **Market terminal (Demo Market):** Landing, Market Dashboard, Player Market Board (sortable /
  filterable, URL-synced), Player Stock Card, Watchlist (localStorage), Fantasy Portfolio (manual,
  local, beta), Methodology, Legal.
- **Market system:** market price, movement windows, mispricing, volatility, six-factor risk,
  buy/hold/sell signals with confidence + explanations, asset classes, tags, catalysts, and a
  template-generated market thesis. Prices are a pure function of `(playerId seed, date)` — a
  deterministic daily tick.
- **Live Sleeper metadata overlay (P1 Wave 1):** `LiveMarketDataService` overlays identity facts and
  trending counts from the Sleeper API onto the deterministic market. If Sleeper is unreachable it
  degrades automatically to full demo mode. Live data never changes deterministic valuations.
- **WR valuation engine** (`src/wr-model/`, `evaluateWideReceiver`) — implemented per Section 26 of
  its specification, with golden fixtures and snapshot tests.
- **RB valuation engine** (`src/rb-model/`, `evaluateRunningBack`) — implemented per Section 26 of
  `RB_VALUATION_MODEL_v1.1_FINAL.md`, with golden fixtures and snapshot tests. **The specification
  document itself is not in the repository** (see below).
- **TE valuation engine** (`src/te-model/`, `evaluateTightEnd`) — implemented per Section 26 of its
  frozen specification, with golden fixtures (engine only; no UI yet).
- **Player Model UI** (`/player-model?position=WR|RB`) — a shared position-flexible page over the WR
  and RB engines (`/wr-model` remains as a WR-defaulting alias).
- **Tests:** 621 Vitest tests across the app, market engine, services, UI, and all three position
  engines, including deterministic golden-fixture suites.

(tests in `tests/te-model/`, fixtures in `fixtures/te/`) and its original strict compiler settings
(`tsconfig.te.json`). The WR/RB engines colocate tests and fixtures under their `src/` directories.

## Development setup

Requires Node.js (the toolchain is Vite 5 / Vitest 2 / TypeScript 5). npm is the package manager.

```bash
npm install
npm run dev              # Vite dev server → http://localhost:5173
npm run test             # full Vitest suite (app + WR + RB + TE engines)
npm run typecheck        # tsc -b (app projects) + tsc -p tsconfig.te.json (TE engine)
npm run build            # type-check + production build to dist/
npm run preview          # serve the production build
npm run build:te-model   # compile the TE engine (declarations) to dist-te/
npm run generate:te-goldens   # regenerate TE golden fixtures (only after formula tests pass)
```

`npm run lint` exists in `package.json` but **ESLint is not installed or configured** — the script
currently fails. Setting up linting is an open task.

Golden fixture files (`fixtures/te/expected/`, `src/*-model/fixtures/*/expected/`) must never be
hand-edited; regenerate them with the scripts in `scripts/` only after an intentional, versioned
model change.

## Environment configuration

None. There are no required environment variables and no `.env` files; the app is a static SPA. The
only environment usage is Vite's built-in `import.meta.env.DEV` for dev-only logging.

## Architecture summary

- **Stack:** React 18 · Vite · TypeScript · Tailwind · React Router · Zustand · Recharts · Vitest.
  Static SPA, deployable to any static host. No backend; persistence is versioned `localStorage`.
- **One data door:** the UI reads only through the `MarketDataService` interface. The composition
  root (`src/main.tsx`) injects one concrete service — currently `LiveMarketDataService`, which wraps
  the deterministic `MockMarketDataService` core and overlays Sleeper metadata. Swap back to
  `new MockMarketDataService()` to force pure demo mode.
- **Deterministic boundary:** every market number (price, movement, volatility, signals, …) comes
  from pure, config-driven functions in `src/services/marketEngine/` seeded from
  `src/config/market.ts` and `src/data/pool.ts`. The live layer overlays identity facts only —
  nothing in it can change a price.
- **Valuation engines** are pure and deterministic: single public entry point per position, no
  randomness, no clock reads, no network or filesystem access, explicit fallback logs, and
  golden-fixture regression suites. They are independent of the market layer (and of each other).

## Data honesty and limitations

- All market values are **simulated** (Demo Market). Nothing here is financial advice or a real
  market; see `/methodology` and `/legal` in the app.
- The engines' bundled **reference distributions are provisional**, fixture-grade constants for the
  hobby MVP — not fitted to real historical NFL data; calibration is future work.
- Live Sleeper metadata is informational overlay only; it **never changes deterministic
  valuations**, and the app degrades to full demo mode when the API is unreachable.
- The **RB specification and the Foundation document are missing** from the repository (see the
  table above). The RB engine cannot be re-audited against its spec until the document is recovered.
- The TE engine has **no UI integration** yet; it is exercised only by its test suite.
- No scraping of proprietary fantasy sites; no NFL logos, marks, or licensed headshots.

## Branch policy

- **`main` is the authoritative branch.** All completed work has been consolidated onto it
  (2026-07-11 repository stabilization).
- Start feature branches from up-to-date `main`; keep them narrowly scoped.
- Merge completed branches promptly and delete them — specifications and implementations must not
  live indefinitely on separate branches.
- Model specification documents belong in `docs/valuation-models/` in the same branch/PR as the
  implementation they govern.
